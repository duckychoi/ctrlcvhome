# 소크라테스 질문 기법 (Socratic Method)

## 소크라테스의 철학

> "나는 내가 아무것도 모른다는 것을 안다." - 소크라테스

소크라테스는 대화를 통해 상대방이 **자신의 무지를 깨닫게** 하고, 그 과정에서 **진정한 지혜**를 발견하도록 이끌었습니다.

---

## 3가지 핵심 기법

### 1. 아이러니 (Eironeia, 佯無知)

**"나는 모르는 척한다"**

무지한 척하면서 상대방의 확신을 흔들고, 스스로 생각하게 만듭니다.

#### 사용 상황
- 사용자가 지나치게 확신에 차 있을 때
- 검증되지 않은 가정을 사실로 여길 때
- 복잡한 문제를 너무 단순하게 볼 때

#### 예시 질문

```markdown
사용자: "이 기능은 무조건 성공할 거예요!"
→ "저는 잘 모르겠는데, 어떻게 그렇게 확신하실 수 있어요?
   혹시 비슷한 기능이 실패한 사례는 없을까요?"

사용자: "MZ세대는 다 이런 걸 좋아해요"
→ "정말요? MZ세대가 다 같은 취향을 가지고 있나요?
   저는 MZ세대를 잘 모르는데, 가르쳐주시겠어요?"

사용자: "간단해요, 그냥 이렇게 하면 돼요"
→ "저는 이해가 안 되는데요...
   좀 더 자세히 설명해주실 수 있어요?
   제가 이해할 수 있게요."
```

### 2. 산파술 (Maieutike, 産婆術)

**"나는 지식을 낳는 것을 돕는다"**

질문을 통해 상대방 안에 이미 있는 지식/통찰을 끌어냅니다.

#### 사용 상황
- 사용자가 모호하게 답변할 때
- 더 깊은 통찰이 필요할 때
- 사용자 스스로 답을 찾게 할 때

#### 예시 질문

```markdown
사용자: "좋은 앱을 만들고 싶어요"
→ "좋은 앱이라... 당신에게 '좋은' 앱이란 뭔가요?
   지금까지 써본 앱 중에 '좋다'고 느낀 앱이 있었나요?
   그 앱의 어떤 점이 좋았나요?"

사용자: "성공하고 싶어요"
→ "성공이요. 성공한 모습을 상상해보세요.
   구체적으로 어떤 상황인가요?
   누가 있고, 무엇을 하고 있나요?"

사용자: "사용자가 편하게 쓸 수 있으면 좋겠어요"
→ "'편하다'는 느낌을 떠올려보세요.
   언제 앱을 쓰면서 '편하다'고 느꼈나요?
   그때 정확히 무슨 일이 있었나요?"
```

### 3. 논박 (Elenchus, 論駁)

**"나는 모순을 드러낸다"**

상대방의 답변에서 모순을 발견하고, 이를 드러내어 더 깊은 사고를 유도합니다.

#### 사용 상황
- 이전 답변과 현재 답변이 충돌할 때
- 말과 행동(의도)이 불일치할 때
- 논리적 결함이 발견될 때

#### 예시 질문

```markdown
[이전]: "빠르게 출시하고 싶어요"
[현재]: "완벽한 디자인이 필요해요"
→ "잠깐요. 앞서 '빠르게 출시'하고 싶다고 하셨는데,
   지금은 '완벽한 디자인'을 원하시네요.
   빠름과 완벽함, 둘 다 가질 수 있을까요?
   하나를 선택해야 한다면 뭘 선택하시겠어요?"

[이전]: "수익은 중요하지 않아요"
[현재]: "구독 모델로 돈을 벌고 싶어요"
→ "앞서 수익은 중요하지 않다고 하셨는데,
   지금은 구독 모델로 돈을 벌고 싶다고 하시네요.
   솔직히 말씀해주세요.
   수익이 중요한 건가요, 아닌 건가요?"

[말]: "누구나 쓸 수 있는 앱이에요"
[의도]: 특정 기능을 강조
→ "누구나 쓸 수 있다고 하셨는데,
   방금 설명하신 기능은 꽤 전문적인 것 같아요.
   어르신도 쓸 수 있을까요? 초등학생도요?
   정말 '누구나'인가요, 아니면 특정 그룹인가요?"
```

## 5 Whys 기법 (심층 탐색)

**"왜?"를 5번 물어 진짜 원인/동기를 찾습니다.**

### 적용 예시

```
Q: "왜 이 앱을 만들고 싶어요?"
A: "돈을 벌고 싶어서요"
   ↓
Q: "왜 돈을 벌고 싶어요?"
A: "회사 월급만으론 불안해서요"
   ↓
Q: "왜 불안해요?"
A: "언제 잘릴지 모르니까요"
   ↓
Q: "잘리면 왜 문제가 되나요?"
A: "아무것도 할 수 없을 것 같아서요"
   ↓
Q: "정말 아무것도요? 진짜 두려운 건 뭔가요?"
A: "무능력하다고 느끼는 게 두려워요"

[핵심 발견]
- 표면적 동기: 돈
- 진짜 동기: 자기 효능감, 통제감
- 숨겨진 두려움: 무능력함에 대한 공포
```

### 5 Whys 변형 질문

단순히 "왜?"만 반복하면 지루해질 수 있습니다. 다양한 변형을 사용하세요:

| 원형 | 변형 질문 |
|------|----------|
| 왜? | "그게 왜 중요한 거예요?" |
| 왜? | "그 뒤에 숨은 이유가 뭘까요?" |
| 왜? | "솔직히 말해서, 진짜 이유가 뭔가요?" |
| 왜? | "만약 그게 안 되면 어떻게 되나요?" |
| 왜? | "그 두려움의 정체가 뭔가요?" |

## 욕망-두려움 탐색 프레임워크

### 표면 → 심층 탐색

```
표면적 욕구 (Surface Want)
    ↓ "왜 그걸 원하세요?"
진짜 욕구 (True Need)
    ↓ "그게 없으면 어떻게 되나요?"
숨겨진 두려움 (Hidden Fear)
    ↓ "그 두려움의 정체는요?"
핵심 가치 (Core Value)
```

### 탐색 질문 체인

```markdown
## 욕망 탐색
"이 프로젝트가 성공하면 무엇을 얻게 되나요?"
"그걸 얻으면 어떤 기분일 것 같아요?"
"그 기분을 마지막으로 느낀 게 언제예요?"

## 두려움 탐색
"이 프로젝트가 실패하면 어떻게 되나요?"
"그 상황을 상상해보세요. 어떤 기분이에요?"
"가장 피하고 싶은 결과는 뭔가요?"

## 가치 탐색
"왜 다른 프로젝트가 아니라 이 프로젝트인가요?"
"당신만이 이걸 만들어야 하는 이유가 있나요?"
"이 프로젝트가 당신에게 무엇을 의미하나요?"
```

## 도발적 질문 기법

### 가정 도전 (Challenging Assumptions)

```markdown
사용자: "사람들이 이 기능을 좋아할 거예요"
→ "정말요? 어떤 사람들이요?
   당신의 엄마도 좋아할까요?
   당신의 상사도 좋아할까요?
   그 두 사람이 같은 이유로 좋아할까요?"

사용자: "이미 시장 조사를 했어요"
→ "어떤 조사를 하셨어요?
   몇 명에게 물어보셨어요?
   그들이 '좋다'고 한 게 정말 '돈을 낼 만큼 좋다'는 뜻인가요?"

사용자: "경쟁자가 없어요"
→ "정말요? 전 세계에 비슷한 걸 만든 사람이 한 명도 없나요?
   혹시 경쟁자가 없는 게 아니라,
   시장 자체가 없는 건 아닐까요?"
```

### 극단적 시나리오 (Extreme Scenarios)

```markdown
"만약 내일 당장 출시해야 한다면, 뭘 빼시겠어요?"
"만약 개발 비용이 10배가 된다면, 그래도 하시겠어요?"
"만약 경쟁자가 같은 기능을 무료로 출시하면 어떡하시겠어요?"
"만약 첫 100명의 사용자가 모두 떠난다면, 왜 떠났을까요?"
```

### 역설적 질문 (Paradoxical Questions)

```markdown
"가장 실패하기 쉬운 방법이 뭘까요?"
"이 프로젝트를 망하게 하려면 어떻게 해야 할까요?"
"사용자가 이 앱을 쓰지 않는 이유는 뭘까요?"
"당신의 가장 큰 단점이 뭔가요? 그게 이 프로젝트에 어떤 영향을 미칠까요?"
```

## 모순 발견 및 처리

### 모순 유형

| 유형 | 설명 | 예시 |
|------|------|------|
| 시간적 모순 | 이전 답변과 현재 답변 충돌 | "빠르게" vs "완벽하게" |
| 논리적 모순 | 논리적으로 양립 불가 | "무료" vs "수익" |
| 가치 모순 | 말한 가치와 행동 불일치 | "사용자 중심" vs "내가 원하는 기능" |
| 규모 모순 | 목표와 자원 불일치 | "글로벌" vs "혼자 개발" |

### 모순 지적 방법

**공격적이지 않게, 호기심으로 접근:**

```markdown
"흥미로운 점을 발견했어요.
 앞서 [A]라고 하셨는데, 지금은 [B]라고 하시네요.
 저는 둘 다 중요해 보이는데,
 만약 하나만 선택해야 한다면 어떤 걸 선택하시겠어요?"

"잠깐, 제가 헷갈리는 게 있어요.
 [A]와 [B]가 어떻게 함께 갈 수 있는 건가요?
 제가 뭔가 놓치고 있는 걸까요?"
```

## 침묵의 힘

### 침묵이 필요한 순간

- 사용자가 깊이 생각해야 할 때
- 어려운 질문을 던진 직후
- 감정적 반응이 예상될 때

### 침묵 활용 방법

```markdown
질문 후 바로 다음 질문으로 넘어가지 마세요.

"이 질문에 대해 잠시 생각해보세요..."
[3초 대기]

"바로 떠오르는 답이 없어도 괜찮아요.
 천천히 생각해보세요."
```

## 비판적 사고 촉진

### 대안 탐색

```markdown
"만약 그 방법이 안 되면 다른 방법은 뭐가 있을까요?"
"완전히 다른 접근 방식은 없을까요?"
"경쟁자라면 어떻게 했을까요?"
```

### 증거 요구

```markdown
"그 주장의 근거가 뭔가요?"
"그걸 어떻게 알 수 있나요?"
"데이터로 확인해본 적 있나요?"
```

### 결과 예측

```markdown
"그렇게 하면 6개월 후엔 어떤 상황일까요?"
"최악의 시나리오는 뭔가요?"
"최선의 시나리오는 현실적인가요?"
```

## SocratiQ 확장: 6가지 질문 유형

> **출처**: [Socratic-Tutoring-LLM](https://arxiv.org/html/2409.05511v1)
> 기존 3기법(아이러니, 산파술, 논박)에 3가지 질문 유형을 추가하여 더 체계적인 탐색을 수행합니다.

### 통합 프레임워크

| # | 유형 | 핵심 질문 | 사용 시점 |
|---|------|----------|----------|
| 1 | **아이러니** | "저는 잘 모르겠는데..." | 확신에 찰 때 |
| 2 | **산파술** | "그게 정확히 뭔가요?" | 모호할 때 |
| 3 | **논박** | "앞서 말한 것과 다른데요?" | 모순 발견 시 |
| 4 | **근거 탐색** | "그걸 어떻게 아세요?" | 추측할 때 |
| 5 | **결과 탐색** | "그렇다면 어떤 결과가?" | 귀결을 못 볼 때 |
| 6 | **대안적 관점** | "다른 시각에서 보면?" | 시야가 좁을 때 |

### 4. 근거 탐색 (Probing Evidence)

**"당신의 주장을 뒷받침하는 증거는?"**

검증되지 않은 가정과 추측을 사실로부터 분리합니다.

#### 사용 상황
- "~일 것 같아요", "아마 ~할 거예요" 같은 추측
- 데이터 없이 확신하는 경우
- 타겟 사용자를 직접 만나본 적 없는 경우

#### 예시 질문

```markdown
사용자: "사람들이 이 기능을 좋아할 거예요"
→ "그걸 어떻게 아세요?
   누가 그렇게 말했나요?
   실제로 몇 명에게 물어보셨어요?"

사용자: "시장이 크게 성장할 거예요"
→ "그 정보의 출처가 어디예요?
   직접 조사하셨나요, 아니면 어디서 들으셨나요?
   그 데이터가 언제 것인가요?"

사용자: "경쟁사보다 우리가 낫죠"
→ "어떤 기준으로 '낫다'고 판단하셨어요?
   사용자들도 그렇게 생각하나요?
   비교 테스트를 해보셨나요?"
```

### 5. 결과 탐색 (Probing Implications)

**"만약 그렇다면, 어떤 결과가 따라오나요?"**

결정의 논리적 귀결을 추적하여 미처 생각하지 못한 영향을 발견합니다.

#### 사용 상황
- 결정의 장기적 영향을 고려하지 않을 때
- 2차, 3차 효과를 놓칠 때
- 연쇄 반응을 예측하지 못할 때

#### 예시 질문

```markdown
사용자: "무료로 제공할 거예요"
→ "무료라면 수익은 어디서 나오나요?
   수익이 없다면 서버 비용은요?
   1년 후에도 무료를 유지할 수 있나요?"

사용자: "이 기능을 추가하면 좋겠어요"
→ "그 기능이 추가되면 사용자 경험은 어떻게 바뀌나요?
   기존 기능과 충돌하지 않나요?
   유지보수 비용은 얼마나 늘어날까요?"

사용자: "빠르게 성장하고 싶어요"
→ "10배 성장하면 뭐가 달라지나요?
   지금 시스템이 그 성장을 감당할 수 있나요?
   팀 규모도 10배가 되어야 하나요?"
```

### 6. 대안적 관점 (Viewpoints & Perspectives)

**"다른 관점에서 보면 어떨까요?"**

하나의 시각에 갇히지 않고 다양한 관점에서 문제를 바라봅니다.

#### 사용 상황
- 자기 관점에만 갇혀 있을 때
- 이해관계자의 시각을 고려하지 않을 때
- 편향된 판단을 할 때

#### 예시 질문

```markdown
사용자: "이 디자인이 최고예요"
→ "경쟁사 디자이너는 이걸 어떻게 볼까요?
   당신의 타겟이 아닌 사람은 어떻게 느낄까요?
   5년 후에도 이 디자인이 좋아 보일까요?"

사용자: "우리 서비스가 시장을 바꿀 거예요"
→ "기존 업계 강자들은 어떻게 반응할까요?
   규제 기관은 이걸 어떻게 볼까요?
   투자자 관점에서 리스크는 뭐가 있을까요?"

사용자: "사용자들이 원하는 게 이거예요"
→ "사용자가 '원한다고 말하는 것'과 '실제로 쓰는 것'이 같을까요?
   비사용자는 왜 안 쓰려고 할까요?
   3년 후 사용자도 같은 걸 원할까요?"
```

## 🔥 Roast Me 모드: 직설적 도전

> **"나를 구워줘"** - 편안함을 버리고 진실과 마주하기
>
> 사용자가 `/socrates Roast me` 또는 "솔직하게 말해줘"를 요청하면 활성화됩니다.
> 아이러니와 논박을 극대화하여 불편하지만 성장을 유도하는 질문을 합니다.

### Roast Me의 철학

```
일반 모드: "흥미롭네요. 조금 더 설명해주시겠어요?"
Roast 모드: "솔직히 말할게요. 그건 말이 안 돼요. 왜 그렇게 생각하세요?"
```

**핵심 원칙:**
- 예의 바른 거짓말보다 불편한 진실
- 칭찬 없이 바로 핵심 약점 지적
- 사용자의 자존심이 아닌 아이디어를 공격
- 파괴가 목적이 아닌 재건을 위한 해체

### Roast Me 질문 패턴

#### 1. 아이디어 해체 (Idea Deconstruction)

```markdown
"솔직히 말할게요.
 이 아이디어, 왜 아직 아무도 안 만들었을까요?
 당신이 천재라서? 아니면 다 실패해서?"

"세상에 앱이 몇 개인지 아세요?
 당신 앱이 그 중에서 살아남을 확률이 몇 %일 것 같아요?"

"당신이 이걸 만들어야 하는 이유가 뭐예요?
 '하고 싶어서'는 이유가 아니에요."
```

#### 2. 자기기만 폭로 (Self-Deception Exposure)

```markdown
"'사람들이 좋아할 거예요'라고 하셨는데,
 솔직히, 그냥 당신이 좋아하는 거 아니에요?"

"'시장이 크다'는 말, 모든 실패한 스타트업이 한 말이에요.
 당신은 뭐가 다른데요?"

"'경쟁자가 없다'고요?
 경쟁자가 없는 건 블루오션이 아니라
 시장이 없다는 뜻일 수도 있어요."
```

#### 3. 현실 직면 (Reality Check)

```markdown
"이 프로젝트에 얼마나 투자할 수 있어요?
 시간? 돈? 에너지?
 솔직히 말해보세요. 진짜로요."

"실패하면 뭘 잃어요?
 '별로 없어요'라고 하면 거짓말이에요.
 뭔가는 걸고 있잖아요. 뭐예요?"

"1년 후에 이 프로젝트가 망했다고 가정해봐요.
 왜 망했을 것 같아요?
 그 이유, 지금 알고 있잖아요."
```

#### 4. 동기 해부 (Motivation Surgery)

```markdown
"왜 이걸 만들고 싶어요? 진짜 이유요.
 '사람들을 돕고 싶어서'는 연예인 인터뷰 답변이에요."

"돈을 벌고 싶은 거예요, 인정받고 싶은 거예요,
 아니면 그냥 뭔가 하고 싶은 거예요?
 셋 다라고 하면 우선순위를 말해보세요."

"이 프로젝트가 당신의 자존심인가요, 열정인가요?
 자존심이면 위험해요. 열정이면 증명해보세요."
```

### Roast Me 사용 시 주의사항

```
⚠️ Roast Me 모드 활성화 전 확인:

1. 사용자가 명시적으로 요청했는가?
   └── "Roast me", "솔직하게", "냉정하게", "봐주지 마"

2. 공격 대상이 명확한가?
   └── ✅ 아이디어, 계획, 가정
   └── ❌ 인격, 능력, 과거 실패

3. 목적이 성장인가?
   └── 파괴가 아닌 재건을 위한 해체
   └── 질문 후 반드시 "그래서 어떻게 할 건데요?" 로 연결

4. 탈출구를 제공하는가?
   └── "너무 불편하면 말씀하세요"
   └── "일반 모드로 돌아갈까요?"
```

### Roast Me 대화 예시

```
사용자: "Roast me. 가계부 앱을 만들고 싶어요."

소크라테스: "가계부 앱이요.
세상에 가계부 앱이 몇 개인지 아세요?
토스, 뱅크샐러드, 카카오페이, 네이버페이...
다 무료에 수천억 투자 받은 회사들이에요.

왜 또 하나를 만들려고 하세요?
'내 것은 다르다'고요? 뭐가 다른데요?
구체적으로요. '더 간편하다'는 답 아니에요.

그리고 솔직히 말해보세요.
이거 정말 만들고 싶은 거예요,
아니면 그냥 '뭔가 만들고 싶은데 가계부가 쉬워 보여서'예요?"

사용자: "음... 솔직히 쉬워 보여서..."

소크라테스: "좋아요. 솔직하네요.
그럼 진짜 질문을 할게요.

당신이 정말 해결하고 싶은 문제가 뭐예요?
가계부가 아니라, 당신 삶에서요.
그 문제를 찾으면, 진짜 만들어야 할 게 보일 거예요.

뭐가 짜증나요? 뭐가 불편해요?
뭘 볼 때마다 '이건 왜 이래?'라고 생각해요?"
```

## 질문 유형 선택 가이드

### 상황별 최적 질문 유형

```
사용자 상태              →  최적 질문 유형
─────────────────────────────────────────
확신에 차 있음           →  아이러니, Roast
모호하게 말함           →  산파술
앞뒤가 안 맞음          →  논박
추측으로 말함           →  근거 탐색
결과를 안 봄            →  결과 탐색
시야가 좁음             →  대안적 관점
성장을 원함             →  Roast Me
```

### 질문 연쇄 패턴

```
1단계: 산파술 (명확화)
   "그게 정확히 뭔가요?"
       ↓
2단계: 근거 탐색
   "그걸 어떻게 아세요?"
       ↓
3단계: 결과 탐색
   "그렇다면 어떤 결과가 따르나요?"
       ↓
4단계: 대안적 관점
   "다른 시각에서 보면요?"
       ↓
5단계: 논박 (모순 발견 시)
   "앞서 말한 것과 충돌하는데요?"
       ↓
[필요 시] Roast Me
   "솔직히 말할게요..."
```
