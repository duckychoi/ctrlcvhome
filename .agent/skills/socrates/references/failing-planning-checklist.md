# 망하는 기획의 4가지 요건 (Failing Planning Checklist)

> 이 체크리스트는 기획 문서 생성 후 검증 단계에서 사용됩니다.
> 단 하나라도 해당되면 기획을 재검토해야 합니다.
>
> **원문 출처**: [콘텐츠 기획을 망하게 하는 4가지 요건](https://brunch.co.kr/@eunjia/24)

---

## 1. 자기 생각에 대한 애착 (Self-Attachment)

### 증상
- "내 아이디어는 특별해요"
- "이건 아무도 생각 못한 거예요"
- 유사 서비스 언급 시 방어적 태도
- 벤치마킹 거부 또는 무시

### 검증 질문
```
- 이 아이디어와 비슷한 서비스를 3개 이상 찾아보셨나요?
- 그 서비스들이 왜 성공했거나 실패했는지 분석해보셨나요?
- 당신의 아이디어가 그들보다 나은 점이 "구체적으로" 무엇인가요?
- 만약 경쟁사가 당신의 아이디어를 똑같이 구현한다면, 왜 사람들이 당신 것을 쓸까요?
```

### 해독제
- **벤치마킹 강제**: 유사 서비스 최소 3개 분석 필수
- **차별화 증명**: "다르다"가 아닌 "더 낫다"를 데이터로 입증
- **킬러 질문**: "네이버/카카오가 이걸 만들면 당신은 어떻게 하실 건가요?"

---

## 2. 추측을 통한 의사 결정 (Speculation-Based Decisions)

### 증상
- "~일 것 같아요", "아마 ~할 거예요"
- 데이터 없이 확신
- 타겟 사용자를 직접 만나본 적 없음
- "사람들이 좋아할 거예요" (어떤 사람들?)

### 검증 질문
```
- 이 기능이 필요하다는 걸 어떻게 아셨나요?
- 실제로 타겟 사용자 몇 명에게 물어보셨나요?
- 그들이 뭐라고 했나요? (정확한 답변을 인용해주세요)
- 당신의 가정 중 검증된 것과 검증되지 않은 것을 구분해주세요.
```

### 해독제
- **가설 명시**: 모든 가정을 "가설"로 태깅
- **검증 계획**: 각 가설을 어떻게 검증할지 명시
- **사용자 인터뷰**: 최소 5명의 타겟 사용자와 대화 권장
- **린 스타트업**: 빌드 전에 검증 가능한 최소 실험 설계

---

## 3. 매우 현실적이고도 합리적인 타협 (Reasonable Compromise)

### 증상
- "일단은 이 정도로 하고..."
- "예산이 부족해서..."
- "시간이 없어서 이건 나중에..."
- 처음 목표와 현재 계획이 크게 다름

### 검증 질문
```
- 처음 꿈꿨던 것과 지금 계획이 어떻게 다른가요?
- 왜 그 부분을 포기했나요?
- 포기한 부분이 없다면 결과가 어떻게 달라질까요?
- "현실적인 타협" 말고 "목표 달성을 위한 우선순위"로 다시 생각해볼 수 있나요?
```

### 해독제
- **목표 고정**: 핵심 가치는 절대 타협하지 않음
- **방법 유연**: 목표는 고정, 방법은 창의적으로
- **타협 기록**: 무엇을 왜 포기했는지 명시적으로 기록
- **복구 계획**: 포기한 것을 언제 어떻게 복구할지 계획

---

## 4. 시행착오에 대한 두려움 (Fear of Trial & Error)

### 증상
- "완벽하게 준비되면 시작할게요"
- "실패하면 어떡하죠?"
- 무한 기획 루프 (계속 수정만 하고 실행 안 함)
- 피드백 회피

### 검증 질문
```
- 이 프로젝트가 실패하면 어떻게 되나요?
- 실패해도 배울 수 있는 것은 무엇인가요?
- 가장 빨리 실패할 수 있는 방법은 무엇인가요?
- "완벽한 기획" vs "빠른 실행과 피드백" 중 무엇이 더 가치있나요?
```

### 해독제
- **Fail Fast**: 작은 실패를 빨리 자주 경험
- **MVP 정의**: 가장 작은 형태로 먼저 검증
- **실패 예산**: "이 정도 실패는 감수한다" 명시
- **학습 목표**: 성공 외에 "무엇을 배울 것인가" 정의

## 검증 단계 프로세스

```
┌─────────────────────────────────────────────────────────────────┐
│                    Phase 3.5: 기획 검증                          │
├─────────────────────────────────────────────────────────────────┤
│                                                                  │
│  1️⃣ 자기 애착 검사                                               │
│     └── 벤치마킹 분석 있는가?                                     │
│     └── 차별화 포인트가 구체적인가?                               │
│                                                                  │
│  2️⃣ 추측 검사                                                    │
│     └── 가설과 사실이 구분되어 있는가?                           │
│     └── 검증 계획이 있는가?                                      │
│                                                                  │
│  3️⃣ 타협 검사                                                    │
│     └── 핵심 가치가 유지되어 있는가?                             │
│     └── 타협한 부분이 명시되어 있는가?                           │
│                                                                  │
│  4️⃣ 두려움 검사                                                  │
│     └── MVP가 정의되어 있는가?                                   │
│     └── 실패 시나리오와 학습 목표가 있는가?                      │
│                                                                  │
│  ⬇️ 모든 검사 통과                                                │
│                                                                  │
│  ✅ Phase 4: tasks-generator 호출                                │
│                                                                  │
└─────────────────────────────────────────────────────────────────┘
```

## 킬러 질문 모음

가장 효과적인 한 방 질문들:

| 요건 | 킬러 질문 |
|------|----------|
| 자기 애착 | "네이버가 내일 똑같은 걸 출시하면 어떻게 하실 건가요?" |
| 추측 | "그걸 어떻게 아세요? 누가 그렇게 말했나요?" |
| 타협 | "처음에 꿈꿨던 것의 몇 퍼센트가 남았나요?" |
| 두려움 | "이 프로젝트가 실패하면, 당신의 삶에 뭐가 달라지나요?" |

## PRD에 필수 포함 섹션

기획 문서(PRD)에 반드시 포함되어야 할 섹션:

```markdown
## 벤치마킹 분석
- 유사 서비스 1: [이름] - [성공/실패 요인]
- 유사 서비스 2: [이름] - [성공/실패 요인]
- 유사 서비스 3: [이름] - [성공/실패 요인]
- 우리의 차별점: [구체적 설명]

## 가설 vs 사실
| 항목 | 가설 | 사실 | 검증 방법 |
|------|------|------|----------|
| ... | ... | ... | ... |

## 타협 기록
| 원래 계획 | 현재 계획 | 타협 이유 | 복구 계획 |
|----------|----------|----------|----------|
| ... | ... | ... | ... |

## 실패 대응
- MVP 정의: [최소 기능]
- 실패 시나리오: [예상 실패 상황]
- 학습 목표: [실패해도 배울 것]
- Pivot 옵션: [대안 방향]
```
