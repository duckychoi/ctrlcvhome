# 기능 → 화면 매핑 가이드

> 이 문서는 소크라테스 Phase 2에서 사용합니다.
> 핵심 기능을 도출한 후, 그 기능이 화면에 어떻게 구현되는지 집요하게 질문하는 방법을 설명합니다.

---

## 왜 화면 매핑이 중요한가?

### 기존 문제

```
[기획 문서]                    [실제 구현]
"지출 기록 기능"        →      ???
"통계 분석 기능"        →      ???

→ 태스크 생성 시 API, 컴포넌트, 페이지가 따로따로
→ 연결점 검증 없이 개발
→ 결과: Auth 불일치, 404 링크, 데이터 불일치
```

### 개선된 접근

```
[기획 문서]                          [실제 구현]
"지출 기록 화면"                →    /expenses/new
  - 금액 입력 필드                    - AmountInput 컴포넌트
  - 카테고리 선택                     - CategorySelect 컴포넌트
  - 저장 버튼 → 목록으로              - POST /api/expenses
                                      - redirect to /expenses

→ 화면 단위로 태스크 생성
→ 연결점이 명확
→ 통합 테스트 가능
```

## 기능 → 화면 매핑 질문 프레임워크

### Level 1: 화면 존재 확인

```markdown
"[기능]을 하려면 어떤 화면이 필요할까요?"
"그 화면의 이름을 뭐라고 부를까요?"
"그 화면에 들어가면 뭐가 제일 먼저 보이나요?"
```

**예시:**
```
기능: 지출 기록
→ "지출 기록을 하려면 어떤 화면이 필요할까요?"
→ 사용자: "입력 화면이요"
→ "입력 화면이라고 부를게요. 들어가면 뭐가 먼저 보이나요?"
```

### Level 2: 화면 구성 요소

```markdown
"그 화면에 어떤 것들이 있나요?"
"버튼은 몇 개인가요? 각각 뭘 하나요?"
"입력할 곳은 몇 개인가요?"
"목록이 있다면 한 항목에 뭐가 보이나요?"
```

**예시:**
```
화면: 지출 입력 화면
→ "입력할 것들이 뭐가 있나요?"
→ 사용자: "금액, 카테고리, 메모요"
→ "금액은 숫자만 입력하나요? 원 단위가 붙나요?"
→ "카테고리는 목록에서 고르나요? 직접 입력하나요?"
```

### Level 3: 화면 간 연결

```markdown
"이 화면에 어떻게 들어오나요?"
"이 화면에서 어디로 갈 수 있나요?"
"[버튼]을 누르면 어디로 가나요?"
"뒤로 가기를 누르면요?"
```

**예시:**
```
화면: 지출 입력 화면
→ "이 화면에 어떻게 들어오나요?"
→ 사용자: "홈에서 + 버튼을 누르면요"
→ "저장을 누르면 어디로 가나요?"
→ 사용자: "목록 화면으로요"
→ "취소를 누르면요? 뒤로 가나요, 홈으로 가나요?"
```

### Level 4: 예외 상황

```markdown
"데이터가 없으면 이 화면에 뭐가 보이나요?"
"에러가 나면 어떻게 알려주나요?"
"로딩 중에는 뭐가 보이나요?"
"로그인 안 했으면요?"
```

## 화면 유형별 질문 템플릿

### 목록 화면 (List)

```markdown
1. "목록에 한 번에 몇 개가 보이나요?"
2. "더 보려면 어떻게 하나요? 스크롤? 페이지 버튼?"
3. "목록 하나를 누르면 어디로 가나요?"
4. "검색이나 필터가 있나요?"
5. "목록이 비어있으면 뭐가 보이나요?"
6. "정렬 순서는 어떻게 되나요?"
```

### 입력 화면 (Form)

```markdown
1. "어떤 것들을 입력하나요?"
2. "필수 입력은 뭔가요?"
3. "유효성 검사가 필요한 게 있나요?"
4. "저장 버튼은 언제 활성화되나요?"
5. "저장 성공하면 어디로 가나요?"
6. "저장 실패하면 어떻게 알려주나요?"
```

### 상세 화면 (Detail)

```markdown
1. "어떤 정보들이 보이나요?"
2. "수정 버튼이 있나요? 어디에?"
3. "삭제할 수 있나요? 확인 창이 뜨나요?"
4. "공유하기가 있나요?"
5. "관련 항목으로 이동할 수 있나요?"
```

### 대시보드/홈 화면

```markdown
1. "처음 들어오면 뭐가 가장 눈에 띄나요?"
2. "요약 정보가 있나요? 어떤 숫자들?"
3. "바로 이동할 수 있는 곳이 어디어디인가요?"
4. "새로운 알림이나 업데이트가 어떻게 보이나요?"
```

## 화면 매핑 체크리스트

### 각 화면에 대해 확인할 것

| 항목 | 질문 |
|------|------|
| **이름** | 이 화면을 뭐라고 부를까요? |
| **URL** | 주소는 어떻게 되나요? |
| **진입점** | 어디서 어떻게 들어오나요? |
| **주요 요소** | 뭐가 보이나요? (컴포넌트) |
| **데이터** | 어떤 데이터가 필요하나요? |
| **액션** | 뭘 할 수 있나요? (버튼) |
| **이동** | 어디로 갈 수 있나요? |
| **인증** | 로그인이 필요한가요? |

### 화면 간 연결 확인

```
화면 A → 화면 B
  ├─ 어떤 액션으로?
  ├─ 어떤 데이터를 전달?
  └─ 돌아올 수 있나?
```

## 집요한 질문 예시

### 사용자가 "그냥"이라고 답할 때

```markdown
사용자: "그냥 목록이요"

→ "목록이요. 좋아요. 근데 목록이 비어있으면 뭐가 보이나요?"
→ "'아직 데이터가 없습니다'? 아니면 바로 추가 버튼?"
→ "목록 하나를 누르면 상세로 가나요, 바로 수정 모드인가요?"
```

### 사용자가 "당연히"라고 답할 때

```markdown
사용자: "당연히 홈으로 가죠"

→ "홈으로 가는군요. 근데 홈에 뭐가 있나요?"
→ "방금 저장한 거를 홈에서 바로 볼 수 있나요?"
→ "아니면 목록 화면으로 가서 확인해야 하나요?"
```

### 사용자가 "나중에"라고 답할 때

```markdown
사용자: "에러 처리는 나중에요"

→ "에러 화면은 나중에 디자인해도 돼요.
   근데 에러가 나면 사용자가 뭘 할 수 있는지는 지금 정해야 해요.
   다시 시도? 홈으로 가기? 고객센터 연결?"
```

## 화면 매핑 결과 기록

Phase 2 완료 시 다음 형식으로 화면 목록을 정리합니다:

```markdown
## 화면 목록

### 1. 홈 화면 (/)
- **진입**: 앱 시작, 하단 탭
- **요소**: 오늘 지출 요약, 최근 기록 3개, 퀵 액션 버튼
- **이동**: 기록 추가(→/new), 목록(→/list), 분석(→/stats)

### 2. 지출 기록 화면 (/new)
- **진입**: 홈 + 버튼, 플로팅 버튼
- **요소**: 금액 입력, 카테고리 선택, 날짜/시간, 메모
- **액션**: 저장, 취소
- **이동**: 저장 성공 시 → 목록

### 3. 지출 목록 화면 (/list)
- **진입**: 홈, 하단 탭
- **요소**: 날짜별 그룹, 각 항목(금액, 카테고리, 메모 미리보기)
- **필터**: 기간, 카테고리
- **이동**: 항목 클릭 → 상세

### 4. 지출 상세 화면 (/list/:id)
- **진입**: 목록에서 항목 클릭
- **요소**: 전체 정보, 수정/삭제 버튼
- **이동**: 수정 → 수정화면, 삭제 → 목록
```

이 결과가 06-screens.md에 저장되고, /screen-spec의 입력이 됩니다.

---

## /screen-spec과의 연결

소크라테스의 화면 매핑 결과(06-screens.md)는 /screen-spec의 입력입니다.

```
/socrates Phase 2
    ↓
06-screens.md (화면 목록 + 간단한 구성)
    ↓
/screen-spec
    ↓
specs/screens/*.yaml (상세 명세)
    ↓
/tasks-generator
    ↓
TASKS.md (화면 단위 태스크)
```
