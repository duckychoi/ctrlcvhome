# 제안 카드 템플릿

## 제안 카드 구조

```
┌─────────────────────────────────────────────────────────────────┐
│  [번호] 서비스명 - 부제목                    [MVP 추천 배지]    │
├─────────────────────────────────────────────────────────────────┤
│  💡 가치 제안 (한 문장)                                         │
│                                                                 │
│  🎯 주요 기능:                                                  │
│     • 기능 1                                                    │
│     • 기능 2                                                    │
│     • 기능 3                                                    │
│                                                                 │
│  👤 타겟: 타겟 사용자 설명                                      │
│  🆚 차별점: 기존 대비 차별점                                    │
│  📱 예상 화면: N개                                              │
│  ⚡ 복잡도: ●●●○○ 레벨                                         │
└─────────────────────────────────────────────────────────────────┘
```

---

## 필드 상세

### 서비스명

```yaml
format: "영문명 - 한글 설명"
examples:
  - "TripSync - 실시간 여행 협업 플래너"
  - "BookClub - 독서 모임 올인원"
  - "InvoiceNow - 5분 인보이스 발행"

guidelines:
  - 영문명: 2음절 이하 권장, 발음하기 쉽게
  - 한글 설명: 핵심 가치를 담은 부제목
  - 도메인 확인: {name}.com 가용 여부는 나중에
```

### 가치 제안 (💡)

```yaml
format: "한 문장으로 핵심 가치"
template: "{누구}를 위한 {무엇}을 {어떻게} 해주는 서비스"

examples:
  good:
    - "친구들과 실시간으로 여행 일정을 함께 짜는 서비스"
    - "링크 하나로 여행 일정 공유, 앱 설치 불필요"
    - "프리랜서가 5분 만에 전문적인 인보이스를 보내는 서비스"

  bad:
    - "여행 관련 종합 플랫폼" (모호함)
    - "최고의 여행 앱" (차별점 없음)
    - "AI 기반 혁신적인 솔루션" (버즈워드)
```

### 주요 기능 (🎯)

```yaml
count: 3개 (정확히)
format: "동사 + 목적어" 또는 "명사 + 설명"

examples:
  good:
    - "실시간 동시 편집 (Google Docs처럼)"
    - "장소별 투표 기능"
    - "비용 분담 자동 계산"

  bad:
    - "좋은 UX" (추상적)
    - "AI 기능" (모호함)
    - "소셜 기능, 공유 기능, 알림 기능, 설정..." (너무 많음)

priority:
  - 1번: 가장 핵심 기능 (이것만 있어도 가치 있음)
  - 2번: 두 번째 중요 기능
  - 3번: 있으면 좋은 기능
```

### 타겟 사용자 (👤)

```yaml
format: "구체적인 사용자 그룹"

examples:
  good:
    - "그룹 여행을 자주 가는 20-30대"
    - "월 10건 이상 인보이스 발행하는 프리랜서"
    - "10명 이하 소규모 독서 모임"

  bad:
    - "모든 사람" (너무 넓음)
    - "여행자" (너무 모호)
    - "B2B와 B2C 모두" (집중 안 됨)
```

### 차별점 (🆚)

```yaml
format: "차별점 (기존 대비 비교)"

examples:
  good:
    - "실시간 협업 (기존 앱들은 순차적 편집)"
    - "제로 설치 (기존 앱들은 모두 설치 필요)"
    - "한국 계좌 연동 (해외 앱은 미지원)"

  bad:
    - "더 좋은 UX" (측정 불가)
    - "혁신적" (의미 없음)
    - "저렴한 가격" (차별점 아님)
```

### 예상 화면 (📱)

```yaml
format: "N개"
range: 3-15개

complexity_guide:
  "3-4개": "극단적 MVP, 핵심 1개 플로우"
  "5-7개": "기본 MVP, 핵심 2-3개 플로우"
  "8-10개": "확장 MVP, 여러 기능"
  "11+개": "복잡, MVP 아님"

typical_screens:
  - 홈/대시보드
  - 리스트
  - 상세
  - 생성/편집
  - 설정
  - 로그인/회원가입
```

### 복잡도 (⚡)

```yaml
format: "●●●○○ 레벨명"

levels:
  "●○○○○ 간단":
    screens: "3-4개"
    features: "CRUD 기본"
    integration: "없음"
    timeline: "1주"

  "●●○○○ 조금 간단":
    screens: "4-5개"
    features: "기본 + 알림"
    integration: "1개"
    timeline: "1-2주"

  "●●●○○ 보통":
    screens: "5-7개"
    features: "협업/실시간"
    integration: "2-3개"
    timeline: "2-3주"

  "●●●●○ 복잡":
    screens: "8-10개"
    features: "양면마켓/결제"
    integration: "4개+"
    timeline: "4주+"

  "●●●●● 매우 복잡":
    screens: "11+개"
    features: "멀티플랫폼"
    integration: "많음"
    timeline: "2개월+"
```

## MVP 추천 배지

```yaml
badge: "⭐ MVP 추천"

criteria:
  - 복잡도: ●●○○○ 이하
  - 화면 수: 5개 이하
  - 외부 연동: 1개 이하
  - 기술 위험: 낮음

display:
  제목 옆에 배지 표시
  하단에 "가장 빠르게 만들 수 있는 건 [N] {이름}입니다." 추가
```

## 제안 세트 구성

### 3개 제안 (기본)

```yaml
proposal_1:
  direction: "차별화 특화"
  complexity: "보통"
  risk: "중간"

proposal_2:
  direction: "기능 확장"
  complexity: "복잡"
  risk: "높음"

proposal_3:
  direction: "극단적 단순"
  complexity: "간단"
  risk: "낮음"
  badge: "⭐ MVP 추천"
```

### 4개 제안 (심층)

```yaml
proposal_1: "기술 차별화"
proposal_2: "시장 확장"
proposal_3: "틈새 특화"
proposal_4: "극단적 단순 ⭐"
```

## eureka-proposal.md 생성

사용자 선택 후 생성되는 문서:

```markdown
# Eureka Proposal: {서비스명}

## 선택된 제안

| 항목 | 내용 |
|------|------|
| 서비스명 | {이름} |
| 가치 제안 | {한 문장} |
| 타겟 | {타겟 사용자} |
| 차별점 | {차별점} |

## 주요 기능

1. **{기능1}**: {설명}
2. **{기능2}**: {설명}
3. **{기능3}**: {설명}

## 예상 화면

| 화면 | 설명 | 우선순위 |
|------|------|----------|
| 홈 | 메인 대시보드 | 필수 |
| ... | ... | ... |

## MVP 범위

- 예상 화면: {N}개
- 복잡도: {레벨}
- 외부 연동: {목록}

## 다음 단계

이 문서를 기반으로:
1. /screen-spec 실행
2. /tasks-generator 실행
```
